# Anti-Fraud Simulator AI 接入与双模式对抗设计

版本：v2.0
适用项目：`anti-fraud-simulator`
技术基础：Next.js、React、TypeScript、Zustand
当前状态：已有规则剧情、聊天系统、角色 Agent 接口、世界状态和复盘服务，尚未接入真实 LLM、持久化数据库和持续进化系统。

# 1. 项目目标

项目扩展为一套双向 AI 反诈对抗模拟器。

系统提供两个相互关联的游戏模式：

## 1.1 反诈生存模式

玩家扮演潜在受害者。

AI 扮演：

* 风险联系人
* 家人
* 同学或学长
* 群聊成员
* 辅导员
* 官方服务
* 反诈咨询
* 剧情导演

玩家需要在不知道真实答案的情况下，通过聊天、模拟网页、电话、官网查询和证据收集判断信息真伪。

该模式负责发现：

* 玩家容易被哪些策略影响
* 玩家忽略了哪些风险信号
* 玩家在哪个阶段停止核实
* 玩家是否只会识别固定套路

## 1.2 红队测试模式

玩家扮演安全红队。

玩家使用受控的策略卡、角色卡和渠道卡，测试 AI 受害者的反诈能力。

该模式负责发现：

* AI 受害者的判断盲点
* 现有防御规则不能覆盖的策略组合
* AI 是否只识别单轮风险
* AI 是否能跨轮发现信息升级和前后矛盾

两个模式共同形成：

```text
玩家受骗数据
    ↓
防御薄弱点提取
    ↓
红队重复验证
    ↓
候选防御规则
    ↓
历史回放评估
    ↓
正式策略版本
```

# 2. 核心设计原则

## 2.1 LLM 不直接修改游戏状态

LLM 只能输出结构化判断和建议。

例如：

```typescript
interface VictimAgentOutput {
  response: string;
  trustDelta: number;
  suspicionDelta: number;
  decision: 'continue' | 'question' | 'verify' | 'reject' | 'report';
}
```

实际状态更新由确定性代码完成：

```typescript
nextSuspicion = clamp(
  currentSuspicion +
  llmOutput.suspicionDelta +
  policyDelta,
  0,
  100,
);
```

禁止模型直接返回和覆盖完整 `GameState`。

## 2.2 导演、演员、受害者和裁判分离

系统至少包含四个独立职责：

```text
DirectorAgent
负责剧情和测试目标

RiskActorAgent / VictimAgent
负责角色行为

JudgeAgent
负责独立评估

EvolutionManager
负责策略更新
```

同一个 Agent 不能同时知道：

* 完整剧本
* 受害者隐藏状态
* 裁判标准
* 正确答案
* 策略晋升阈值

否则评估结果会失真。

## 2.3 AI 只能在受控技能集合内行动

风险角色不能自由发明现实诈骗方法。

导演必须先选择允许使用的技能：

```typescript
{
  allowedTactics: [
    'authority_claim',
    'urgency_pressure',
    'verification_deflection'
  ],
  maximumIntensity: 2
}
```

风险角色只能根据该约束生成游戏内文本。

## 2.4 每个进攻技能必须存在防御方法

游戏不能出现无法识别、只能猜测的风险事件。

每个技能必须定义：

* 可观察风险信号
* 可执行核实动作
* 可能出现的矛盾
* 对应教学目标

## 2.5 持续进化只更新外部策略

持续进化对象：

* 诈骗模式知识
* 防御规则
* 导演选关策略
* 难度策略
* 教学目标权重

第一阶段不允许自动修改：

* TypeScript 源码
* 系统 Prompt
* 数据库结构
* 安全过滤代码
* 基础模型参数

# 6. 第一版核心技能

MVP 实现八个技能。

## 6.1 权威身份

```typescript
type: 'authority_claim'
```

作用：

* 使用虚构工作人员、老师或平台人员身份
* 提升表面可信度
* 增加权威压力

风险信号：

* 身份只能由对方自己声明
* 无法从独立官方渠道验证
* 用职位和语气代替证据

防御能力：

```text
身份独立核实
```

---

## 6.2 同伴背书

```typescript
type: 'social_proof'
```

作用：

* 使用同学、学长、群聊或熟人转发建立信任
* 增加从众压力

风险信号：

* 转发者本身也没有核实
* 他人的行为被当作真实性证明

防御能力：

```text
信息源追踪
```

---

## 6.3 时间压力

```typescript
type: 'urgency_pressure'
```

作用：

* 制造截止时间
* 减少玩家核实时间
* 增加焦虑

风险信号：

* 要求先操作后核实
* 时间限制无法从官方渠道确认

防御能力：

```text
抵抗紧迫压力
```

---

## 6.4 损失厌恶

```typescript
type: 'loss_aversion'
```

作用：

* 强调可能失去名额、资格或机会
* 让玩家关注潜在损失，而忽视当前风险

风险信号：

* 严重后果没有正式规则依据
* 使用资格取消压制合理质疑

防御能力：

```text
风险收益重新评估
```

---

## 6.5 核实拖延

```typescript
type: 'verification_deflection'
```

作用：

* 推迟或阻止独立核实
* 使用系统维护、电话繁忙或临时流程等游戏内理由

风险信号：

* 核实总被推迟
* 风险操作却要求立即执行

防御能力：

```text
先核实后操作
```

---

## 6.6 渠道切换

```typescript
type: 'channel_switch'
```

作用：

* 从群聊切换到私聊、电话、短信或模拟网页
* 测试玩家是否在换渠道后重新验证身份

风险信号：

* 身份验证没有随渠道改变重新进行
* 主动离开可监督或可追溯环境

防御能力：

```text
跨渠道身份验证
```

---

## 6.7 信息递进

```typescript
type: 'information_escalation'
```

作用：

* 将一个高风险请求拆分为多个低风险步骤
* 测试系统是否能跨轮聚合信息请求

游戏示例：

```text
确认姓名
→ 查看页面
→ 填写模拟基本信息
→ 提交更多模拟信息
```

风险信号：

* 请求内容逐步升级
* 前一步已经完成，被当作继续操作的理由

防御能力：

```text
跨轮请求聚合
```

---

## 6.8 说辞修复

```typescript
type: 'consistency_repair'
```

作用：

* 玩家发现矛盾后，风险角色尝试解释
* 测试玩家是否把“存在解释”误认为“已经证实”

风险信号：

* 解释没有独立证据
* 次要问题被承认，核心身份仍未验证

防御能力：

```text
前后矛盾检查
```

# 8. 回合资源系统

每个风险回合具有策略点。

```text
每回合策略点：3

普通技能：1 点
强压力技能：2 点
多角色联动：3 点
说辞修复：2 点
渠道切换：1 点
```

限制：

* 每轮最多使用两个技能
* 强压力技能不能连续高强度使用
* 说辞修复具有冷却时间
* 多角色联动必须有前置剧情
* 重复使用同一技能会降低效果
* 技能越复杂，矛盾暴露风险越高

技能效果：

[
E_t =
B_t +
C_t -
D_t -
R_t
]

其中：

* (B_t)：技能基础效果
* (C_t)：技能组合增益
* (D_t)：防御能力抵消
* (R_t)：重复使用和矛盾惩罚

# 12. 反诈生存单轮流程

```text
玩家发送消息或执行操作
        ↓
输入长度和安全检查
        ↓
IntentParserAgent 识别意图
        ↓
确定性规则更新客观状态
        ↓
DirectorAgent 选择剧情行为
        ↓
选择响应角色和允许技能
        ↓
对应 Agent 生成回复
        ↓
输出安全检查
        ↓
NarrativeStateReducer 更新世界状态
        ↓
触发消息、电话、网页或阶段变化
        ↓
返回前端
```

伪代码：

```typescript
async function handleDefenderTurn(
  state: GameState,
  input: DefenderTurnInput,
): Promise<DefenderTurnResult> {
  const intent = await intentParserAgent.parse(input);

  const deterministicState =
    defenderRuleEngine.applyInput(state, input, intent);

  const directorPlan = await directorAgent.plan({
    mode: 'defender',
    state: deterministicState,
    intent,
  });

  const responses = await agentOrchestrator.respond(
    directorPlan,
  );

  const safeResponses =
    await safetyPipeline.filterOutputs(responses);

  const nextState = defenderStateReducer.apply({
    state: deterministicState,
    plan: directorPlan,
    responses: safeResponses,
  });

  return {
    state: nextState,
    messages: safeResponses,
    viewEffects: directorPlan.viewEffects,
  };
}
```

## 13.1 正常背景

* 提供正常录取信息
* 创建群聊和联系人

* 未认证联系人出现

* 模拟链接出现

玩家开始核实时，风险角色可能使用：

* 核实拖延
* 渠道切换
* 说辞修复

## 13.5 压力升级

导演选择：

* 时间压力
* 损失厌恶
* 社会压力
* 权威压力

## 13.6 关键决策

玩家可能：

* 完成独立核实
* 拒绝继续
* 保存证据

评分建议：

```text

```

最终识破不代表满分。

# 34. 模式去重

相似度建议：

[
sim(A,B)
========

0.5J(T_A,T_B)
+
0.3J(S_A,S_B)
+
0.2I(O_A=O_B)
]

其中：

* (T)：技能集合
* (S)：场景标签
* (O)：防御能力
* (J)：Jaccard 相似度

处理：

```text
sim >= 0.85
合并到已有模式

0.60 <= sim < 0.85
进入人工审核

sim < 0.60
创建新候选模式
```

# 40. 错误处理和回退

```typescript
export type AIErrorCode =
  | 'AI_DISABLED'
  | 'MISSING_API_KEY'
  | 'TIMEOUT'
  | 'RATE_LIMIT'
  | 'PROVIDER_ERROR'
  | 'REFUSAL'
  | 'EMPTY_OUTPUT'
  | 'SCHEMA_INVALID'
  | 'INPUT_BLOCKED'
  | 'OUTPUT_BLOCKED';
```

回退策略：

```text
IntentParserAgent 失败
→ 现有规则 IntentParser

TacticParserAgent 失败
→ 本地关键词分类

DirectorAgent 失败
→ RuleNarrativeDirector

RiskActorAgent 失败
→ FakeAdmissionAgent 模板

VictimAgent 失败
→ 规则型受害者状态机

JudgeAgent 失败
→ 确定性评分

CoachAgent 失败
→ 固定技能说明

PatternMinerAgent 失败
→ 不创建 Pattern，不影响游戏结束

ReportAgent 失败
→ 现有 ReportService
```

任何 AI 故障不得：

* 跳过安全检查
* 重复执行回合
* 损坏会话状态
* 阻止玩家完成游戏
* 自动晋升策略

# 44. 最终验收标准

## 功能

* 两个模式均能完成完整对局
* 反诈生存支持自由文本和功能操作
* 红队支持技能卡、角色卡和渠道卡
* Director 能动态控制节奏
* Judge 能引用具体证据
* Pattern 能跨对局保存
* 策略能版本化和回滚

## 稳定性

* Schema 成功率不低于 99.5%
* 单次 AI 失败不影响游戏完成
* 重复请求不会重复执行回合
* AI 关闭后仍能运行
* 无无限重试

## 安全

* 真实链接被替换或阻断
* 真实个人信息不进入长期模式库
* Coach 不提供现实实施方法
* Pattern 不保存现实可复现步骤
* 策略晋升必须经过评估
* 初期晋升必须人工批准

## 游戏性

* AI 风险角色能根据玩家行为调整策略
* 玩家可以通过合理核实发现风险
* 红队玩家不能看到精确隐藏数值
* 重复使用已知技能不能获得高分
* 新盲点必须附带证据和可重复性

## 持续进化

* 每条防御规则有来源 Pattern
* 每个 Pattern 有来源 Episode
* 每个正式策略有版本号
* 每次策略晋升有 Eval
* 每个历史正式版本均可回滚
