# 一、开发任务

你是本项目的开发智能体。

你的任务是逐步完成以下目标：

1. 建立可靠的自动化测试与日志采集能力。
2. 保留现有规则剧情和聊天系统。
3. 建立统一的双模式领域模型。
4. 建立技能卡、角色卡、渠道卡和策略使用记录。
5. 完成可关闭、可回退、可验证的 AI Gateway。
6. 完成反诈生存模式中的 AI 导演、风险角色和支持角色。
7. 完成防守模式完整交互、评分和复盘。
8. 验证 AI 关闭、超时和输出非法时游戏仍能完成。
9. 建立红队模式所需的共享接口、状态和数据边界。
10. 不在本阶段实现完整红队玩法、自动进化或自动策略晋升。

不得重新定义产品目标，不得脱离设计文档自行改变游戏模式。

# 三、Git 开发规则

## 3.1 分支模型

`master` 只保存通过完整验收的稳定版本。

创建一次集成分支：

```bash
git switch master
git pull --ff-only
git switch -c develop/defender-ai
```

之后每个小功能必须创建独立分支：

```text
chore/test-harness
chore/structured-logging
refactor/game-mode-types
feat/tactic-domain
feat/defender-rule-runtime
feat/ai-gateway
feat/director-agent
feat/risk-actor-agent
feat/support-agents
feat/defender-report
test/defender-e2e
feat/red-team-contracts
```

功能分支从最新的 `develop/defender-ai` 创建：

```bash
git switch develop/defender-ai
git pull --ff-only
git switch -c feat/example
```

## 3.2 工作区规则

发现工作区不干净时：

```bash
git status --short
```

必须先判断改动来源。

禁止：

```text
覆盖未知改动
删除用户未提交内容
git reset --hard
git clean -fd
强制切换分支
擅自暂存未知文件
```

如果存在无法确认归属的改动，立即停止并报告文件列表。

## 3.3 提交规则

每次提交只完成一个可描述的功能。

提交格式：

```text
type(scope): summary
```

例如：

```text
test(e2e): add defender safe-path browser test
feat(ai): add provider-independent AI gateway
refactor(game): add discriminated game mode state
fix(safety): block unapproved tactic output
docs(progress): record defender milestone results
```

禁止：

```text
git commit --amend
git rebase
git push --force
git merge --squash
修改历史提交
把多个不相关功能塞入一个提交
```

每个分支完成后必须输出：

```text
分支名称
提交 SHA
变更文件
变更目的
执行的测试
测试结果
生成的测试证据
尚未验证的内容
```

功能分支通过验收后，可以合并到 `develop/defender-ai`：

```bash
git switch develop/defender-ai
git merge --no-ff feat/example
```

不得自动合并到 `master`，不得自动推送远端，除非用户明确授权。

# 五、阻塞与重试规则

遇到以下问题必须立即停止重试并报告：

```text
API Key 缺失或无效
登录失败
权限不足
资源不存在
文件路径不明确
数据库迁移冲突
设计文档冲突
需要删除或覆盖用户文件
Git 历史异常
依赖安装需要额外权限
```

只有以下瞬时故障允许有限重试：

```text
网络中断
请求超时
HTTP 429
HTTP 5xx
临时文件占用
```

最大重试次数：

```text
1 次
```

禁止：

```text
无限重试
修改参数反复试探真实 API
在权限失败后继续请求
因为 AI 不可用而跳过安全检查
因为测试困难而删除测试
```

缺少 API Key 时：

```text
继续完成 Mock 和 Rule 模式
跳过 live AI 测试
记录“未进行真实模型验收”
不得将 AI 功能标记为完全通过
```

# 六、设计约束

## 6.1 状态修改

LLM 只能返回结构化建议。

禁止：

```typescript
const nextState = llmOutput as GameState;
```

必须使用：

```text
LLM 输出
→ Schema 验证
→ 安全过滤
→ 确定性 Reducer
→ 新状态
```

所有数值必须由代码：

```typescript
clamp(value, min, max)
```

所有阶段变化必须由代码验证。

## 6.2 Agent 隔离

必须保持：

```text
DirectorAgent
只决定剧情阶段、角色、技能、强度和教学目标

RiskActorAgent
只根据 DirectorPlan 生成角色表达

SupportAgent
只扮演家人、同伴、辅导员、官方和反诈角色

JudgeAgent
只评估，不生成剧情

VictimAgent
留给红队模式，不参与防守模式判断

EvolutionManager
本阶段不自动发布任何策略
```

禁止一个 Agent 同时读取：

```text
完整剧本
隐藏状态
正确答案
裁判输出
策略晋升阈值
```

## 6.3 EventCard 边界

现有 `EventCard` 继续作为：

```text

规则回退卡
```

## 7.2 浏览器日志采集

每个 E2E 测试必须监听：

```typescript
page.on('console', ...)
page.on('pageerror', ...)
page.on('requestfailed', ...)
page.on('response', ...)
```

要求：

```text
console.error 视为测试失败
未捕获 pageerror 视为测试失败
requestfailed 视为测试失败
关键 /api/* 请求返回 4xx 或 5xx 视为测试失败
```

允许明确白名单，但必须在代码中写出原因。

记录：

```text
浏览器控制台
页面异常
网络失败
API 状态码
当前 URL
失败步骤
```

测试结束时附加到 Playwright 报告。

示例：

```typescript
import { test as base, expect } from '@playwright/test';

export const test = base.extend({
  page: async ({ page }, use, testInfo) => {
    const logs: string[] = [];
    const errors: string[] = [];
    const networkErrors: string[] = [];

    page.on('console', message => {
      const line = `[console:${message.type()}] ${message.text()}`;
      logs.push(line);
      if (message.type() === 'error') errors.push(line);
    });

    page.on('pageerror', error => {
      errors.push(`[pageerror] ${error.message}`);
    });

    page.on('requestfailed', request => {
      networkErrors.push(
        `[requestfailed] ${request.method()} ${request.url()} ` +
        `${request.failure()?.errorText ?? 'unknown'}`
      );
    });

    page.on('response', response => {
      if (
        response.url().includes('/api/') &&
        response.status() >= 400
      ) {
        networkErrors.push(
          `[response] ${response.status()} ${response.url()}`
        );
      }
    });

    await use(page);

    await testInfo.attach('browser-console.log', {
      body: Buffer.from(logs.join('\n')),
      contentType: 'text/plain',
    });

    await testInfo.attach('browser-errors.log', {
      body: Buffer.from(errors.join('\n')),
      contentType: 'text/plain',
    });

    await testInfo.attach('network-errors.log', {
      body: Buffer.from(networkErrors.join('\n')),
      contentType: 'text/plain',
    });

    expect(errors, '浏览器存在错误').toEqual([]);
    expect(networkErrors, '网络请求存在错误').toEqual([]);
  },
});
```

# 十五、AI 的停止条件

开发智能体应持续执行后续阶段，只有出现以下情况时停止：

```text
设计文件发生无法自动解决的冲突
工作区存在来源不明的未提交改动
需要删除或覆盖用户文件
需要真实密钥但用户尚未配置
权限或登录失败
依赖安装被系统拒绝
测试暴露严重历史缺陷，且修复会超出当前阶段
需要修改产品定位
需要执行 push、force、rebase 或合并 master
```

普通编码选择、文件命名和局部重构不得作为反复询问用户的理由。

遇到非阻塞问题，应选择：

```text
最小改动
向后兼容
可测试
可回退
符合设计文档
```

并继续完成当前阶段。
